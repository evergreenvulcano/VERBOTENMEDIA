# Northstar: Capacity Battery And Quest Layer

## Northstar

This repo is an open editorial capacity battery and a build-game.

It should be able to receive unfamiliar material, protect its voice, run editorial experiments, modulate format, and return useful v0 versions: periodical issues, essays, program texts, article series, concept banks, glossaries, publication objects, pitch surfaces, or other editorial outcomes.

Core sentence:

> Repot ska inte imponera. Repot ska ta emot material och förändra dess möjliga framtid utan att äta upp dess röst.

## Anti-Definition

The repo is not primarily:

- a portfolio
- an archive
- a second brain
- an AI workflow theatre
- a productivity machine
- a consultancy surface
- a generic content platform

It may contain traces of all of those shapes. None of them owns the northstar.

## Utility Principle

Utility prompts are not helpers.

They are the repo's public interface toward other people's material. A utility prompt is where external material can enter the machine without being swallowed by the machine.

The prompt must protect the material's voice before it offers format, structure, or output.

## Voice Shield

Do not eat the source voice.

If the material has its own rhythm, wound, idiom, anger, dryness, comic surface, uncertainty, or field of pressure, preserve it as a living constraint. Do not translate it too quickly into repo language, strategy language, agent language, or clean administrative categories.

## Game Principle

The repo should not speak like an office when it can speak like a playable world.

Game language is not decoration. It is a guard against agents flattening everything into workflows, pipelines, governance, and management diagrams.

No badges. No achievements. No fake gamification.

Use game terms only when they make the work more playable, more discoverable, and less bureaucratic.

## Quest Layer

**GAME**: a larger playable situation in the repo. Examples: Periodical Issue Game, Build Game, XPRMNT-ZN Game, Public Surface Game, Utility Prompt Game.

**QUEST**: a directed run or establishment movement inside a game. A quest does not need a finished task list. Tasks may be discovered during play.

**TASK**: a concrete action discovered, executed, or reformulated during a quest. Do not predefine every task when discovery is the point.

**SIDEQUEST**: a deviation with legitimate possible yield. It may return to the main quest, remain a loose module, or upgrade into its own game.

**Q**: text-as-interface. A Q-text does not only describe work; it opens work. It can be routed through the repo and treated as programming for human/agent action without becoming code or a task list.

**BQ**: Build Quest. Internal work where something is not stable enough to meet public or foreign material yet.

**PQ**: Public Quest. A quest aimed at public-facing reality: entry surface, collaborator entrance, utility prompt, public presentation, or use-the-machine surface.

**SQ**: Side Quest. A deviation that may carry value.

**DQ**: Drive Quest. A quest where Drive is the game board: intake, routing, manifests, sidecars, Codex action, filesystem weirdness.

**IQ**: Integration Quest. A quest where two or more bodies must connect without one eating the other.

**UQ**: Utility Quest. A quest whose drop is a usable prompt, module, machine, or capacity that others can ping material through.

**XQ**: Experimental Quest. A quest where value lives in mutation, playability, test, contamination, or aesthetic engine rather than stability.

Use domain aliases such as `NQ` or `VQ` only when they are already obvious. Do not build a giant abbreviation taxonomy.

## Flip Principle

An internal build quest may flip into a public-facing quest when the body becomes real enough to receive others.

Example:

- `BQ_NORROTIV__establish_the_body.md`
- `PQ_NORROTIV__public_entry_surface.md`

Martin follows the same logic:

- Martin BQ: establish presentation without reducing Martin to CV-person, artist myth, AI consultant, jobseeker, strategy consultant, or project avatar.
- Martin PQ: public-facing presentation where Martin works as an entry point, not the whole work.

## File Naming Syntax

Use clear, playable filenames:

```text
TYPE_DOMAIN__playable_instruction.md
```

Examples:

```text
BQ_NORROTIV__establish_the_body.md
SQ_MARTIN__presentation_without_flattening.md
IQ_HEPTABASE_ESSARUIN__relation_revealing_surface.md
UQ_PERIODICAL_MACHINE__material_to_issue_v0.md
PQ_NORROTIV__public_entry_surface.md
```

Prefer a readable instruction over acronym poetry.

## Guardrail

Do not overbuild.

Do not bureaucratize.

Do not convert game language into municipal process mysticism.

Do not build dashboards, schemas, task managers, JSON control surfaces, CI, or product theatre unless a later quest explicitly requires them.

If the layer starts sounding like an office, lower the abstraction and return to the material.

## Agent Usage

When starting repo work:

1. Identify the current GAME.
2. Name the QUEST type.
3. List starting inventory.
4. Set a voice shield.
5. Choose one loadout/modulator.
6. Make the first move.
7. Let tasks be discovered when discovery matters.
8. Track sidequests without letting them eat the quest.
9. Drop a usable v0 or a clear patch note.

Use this file to orient action, not to block it.
