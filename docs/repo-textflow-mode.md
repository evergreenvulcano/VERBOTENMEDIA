# Repo Textflow Mode v1

## Purpose

Textflow Mode is a narrow operational posture for this repository. It defines the repo as a **publication membrane** rather than a total project archive. The goal is to maintain a focused, legible, and maintainable public-facing channel without attempting to mirror the full volume of working material.

During Textflow Mode, the repo is not the archive. Raw origins, large runs, unstable drafts, visual batches, experimental protocol bodies, and proof material live elsewhere — in OneDrive, Notion, local folders, or other working environments. The repo gives status. It does not contain everything.

---

## What belongs in the repo during Textflow Mode

A file belongs in the repo when it has a clear, present function:

- **Publish a finished text** — essays, literary works, shortstories, fragments with public status
- **Hold a public capsule** — provenance notes, AI-status declarations, editorial rules, method fragments
- **Diagnose publication status** — status files, intake assessments, editorial decisions
- **Protect method** — documents that define how the publication system works
- **Document renderer or publication rules** — specs, standards, system notes
- **Support the static site** — HTML, CSS, JS, templates, and linked assets

No file enters the repo merely because it is interesting. It enters only when it has publication status, diagnostic status, method-protection status, or renderer/support function.

---

## What does not belong in the repo during Textflow Mode

- Raw source exports from Notion, OneDrive, or other working environments
- Large experimental text corpora or agent-protocol runs not yet assigned a publication function
- Visual asset batches not tied to a specific published piece
- Unstable draft clusters that are not yet being actively prepared for publication
- Proof material, version histories, or intermediate working bodies
- Files added because they might be useful later

---

## Inbox limit

The `inbox/writings/` directory is a staging area, not a holding space. Files there are conversion or normalization candidates under active review. Inbox should not accumulate indefinitely. A file in inbox either moves to `static-pages/writings/` as a published piece or is removed from the repo and held in the working archive.

---

## Capsule policy

Capsules live in `static-pages/capsules/`. They are short public-facing knowledge units. They may include:

- Provenance notes
- AI-status declarations
- Editorial rules
- Publication diagnoses
- Method fragments
- Pressure maps

Capsules are not literary prose. They are not substitutes for essays. AI-mediated production is permitted for capsules when the capsule does not claim to be literary prose or direct authorial voice.

---

## AI-status policy

AI may assist with:

- Reading and sorting material
- Status diagnosis
- Summarizing process
- Producing public capsules
- Supporting publication logistics

AI must not:

- Replace literary prose
- Replace authorial voice
- Replace origin-bearing essayistic thought
- Claim to be the writer of work it has not written

Any AI-mediated text that appears publicly must be classifiable as a capsule, log, diagnosis, or synthesis — not as a literary work or direct authorial expression.

---

## Non-goals

Textflow Mode does not:

- Reorganize the repository
- Delete or remove existing creative material
- Flatten or reclassify the full project archive
- Create new agent frameworks or automation systems
- Pretend that the larger body of work does not exist
- Replace manual editorial judgment with automated pipelines

---

## Acceptance criteria for adding new files

Before adding a file to the repo, confirm at least one of the following:

1. It is a finished text being published or staged for publication.
2. It is a capsule with a defined public function.
3. It is a status or diagnostic document serving an active publication decision.
4. It protects or documents a method, standard, or renderer rule.
5. It directly supports the static site infrastructure.

If none of these apply, the file belongs in the working archive, not in the repo.
