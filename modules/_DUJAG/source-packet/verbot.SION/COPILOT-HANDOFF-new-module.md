## Core Design Principles

### 1. Constitutive first

Preserve explicit load-order logic:

1. constitutive source layer first
2. runtime/composition layer second
3. downstream expressive or diagnostic artifacts after that

In practical terms, treat the `Dujag.md`-type surface as prior to runtime composition. Do not flatten constitutive text into a summary note.

### 2. `Dujag.md` as positions file

Do not interpret `Dujag.md` as a normal character instruction, style prompt, roleplay profile, or permission slip to write a character "from the inside."

Its sharper role is positional: the agent is placed in relation to a third-eye/locus condition, a presence-without-presence, and an operative remainder distributed through rooms, objects, silences, artifacts, and other agents' decisions.

### 3. Contracted underinstances

The module should support underinstances as semantically named, methodically delimited, diagnostically useful test bodies.

Each underinstance should carry a semantic contract rather than behaving like a trivial plugin. A contract should state:

- what the underinstance may assume
- what it must retain
- what it may vary
- what its output should diagnostically reveal

### 4. Diagnostic before self-optimization

Do not frame this module as an output-optimization engine. Output may be used as symptom material, but not as the primary truth source for revision.

Preserve the three-part loop:

1. constitutive reading
2. mediated manifestation
3. reflective diagnostic

The module must make it possible to distinguish these three phases cleanly in documentation and usage.

### 5. Protected semantic intensity

Treat certain terms as protected and nontrivial. In particular:

- empathy must not collapse into kindness, warmth, or prosocial reassurance
- empathy requires an intensity note: it is only retained if non-transferability, instability, and uncontrollability remain intact after operation
- third eye / locus language must not collapse into camera perspective, scene mechanics, or mystical ornament
- protected terms must retain enough semantic sovereignty that they are not silently standardized into weaker English defaults

### 6. Separate parallel tracks

If the source material contains distinct bullet-list pipelines, conceptual tracks, or competing initiation lines, do not collapse them into one falsely unified steering surface. The module should allow parallel tracks to coexist when the source packet treats them as separate.

### 7. Sparse but rigorous

Keep the module concentrated. It should be structurally serious but not bloated. Prefer a small set of files with clear responsibilities over a wide framework full of decorative scaffolding.

## Required Conceptual Distinctions

The module must preserve the following distinctions explicitly.

### Positions file vs character instruction

- a positions file places the agent in an operative relation to locus, presence-without-presence, artifacts, silence, and residual agency
- a character instruction tells the agent how to portray or inhabit a character
- `Dujag.md` must be treated as the former before it is ever allowed to influence the latter

### Constitutive vs runtime

- constitutive text establishes conditions of admissibility, retention, and semantic posture
- runtime text shapes downstream composition, activation, and session behavior

### Underinstance vs plugin

- an underinstance carries a semantic contract and produces controlled manifestation or diagnostic material
- a plugin is merely a reusable capability or prompt unit
- do not trivialize underinstances into generic modular helpers

### Expressive vs diagnostic

- expressive outputs are literary or generative manifestations
- diagnostic outputs are tests, failure renderings, contrast pairs, paraphrase checks, drift probes, and similar semantic test surfaces

### Review taxonomy

- semantic self-review tests whether the system still holds its own understanding
- manifestation review assesses what actually appeared
- diagnostic review assesses whether the testing apparatus itself was semantically admissible

### Archive vs active layer

- archived material is retained initiation memory
- active layer material is presently admitted for constitutive or runtime use

Retention is mandatory even when admission is withheld.
