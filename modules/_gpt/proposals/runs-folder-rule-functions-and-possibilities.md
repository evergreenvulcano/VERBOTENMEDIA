# Proposal — runs/ folder: functions, rule models, and possibilities

Status: speculative / non-canonical
Scope: modules/_gpt/

---

## Premise

The current `runs/` folder has been governed by a rule optimized for late-stage curation.

Observed condition:
- low proposal density
- under-materialization of run outputs
- weak conversion from process → staged artifact

This proposal reframes `runs/` not as a filtered archive, but as an operational layer.

---

## Core functions of `runs/`

### 1. Temporal memory

`runs/` acts as chronological system memory.

Each file represents:
- a state of interpretation
- a system configuration under pressure

Function:
- preserve evolution without flattening

---

### 2. Audit surface

Runs are inspectable units.

Function:
- trace how outputs were derived
- compare runs structurally (via run-spec)

---

### 3. Prompt calibration feedback

Runs expose:
- where prompts drift
- where structure breaks

Function:
- allow iterative tightening of agent behavior

---

### 4. Promotion reservoir

Runs are not endpoints.

They are the primary source of:
- candidate texts
- structural ideas
- reusable fragments

Function:
- feed `proposals/`

---

### 5. Recovery layer

If chat fails or context is lost:
- runs provide reconstruction material

Function:
- restore continuity without inference

---

## Alternative rule models

### Model A — Formation phase (recommended current state)

Rule:
- keep all structurally legible runs
- prioritize accumulation over elegance
- promote aggressively to `proposals/`

Effect:
- builds mass
- reveals patterns

---

### Model B — Transitional phase

Rule:
- keep runs with clear structural value
- begin selective promotion
- start identifying redundancy

Effect:
- early shaping

---

### Model C — Mature phase (original rule intent)

Rule:
- keep only high-value runs
- prune aggressively
- proposals become primary surface

Effect:
- refinement

---

## Derived possibilities

If `runs/` is treated as an active layer:

### 1. Run clustering

Group runs by:
- theme
- pressure
- output type

---

### 2. Weekly distillation

Derive:
- synthesis documents
- candidate canon shifts

---

### 3. Automatic promotion patterns

Define triggers such as:
- repeated phrasing
- stable structures

→ auto-stage into `proposals/`

---

### 4. Run-ledger file

Maintain:
- index of runs
- cross-links

---

### 5. Experimental branching

Allow parallel run streams:
- different prompt variants
- different interpretive modes

---

## Constraint

This document is not law.

It does not modify:
- write-policy
- fallback-mode
- run-spec

It exists as a staging surface for evaluating how `runs/` should behave.

---

## Short statement

`runs/` is not a filter.

It is a field of accumulation from which structure emerges.
